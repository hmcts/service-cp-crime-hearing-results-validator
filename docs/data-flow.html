Crime Hearing Results Validator

Data Flow — Request to Response  |  service-cp-crime-hearing-results-validator  |  HMCTS / Crime Common Platform

Overview

This document describes how an HTTP validation request flows through the service — from the incoming POST body to the structured JSON response. The service applies configurable YAML + CEL rules to hearing result data and returns structured errors and warnings. 

HTTP POST /validate

        ▼

        │

Step-by-Step Data Flow

1 HTTP Entry — ValidationController

uk.gov.hmcts.cp.controllers.ValidationController

Receives POST /validate. Deserialises the request body into a DraftValidationRequest and delegates immediately to ValidationService.validate(). No business logic lives here.

2 Orchestration — DefaultValidationService

uk.gov.hmcts.cp.services.impl.DefaultValidationService

  • Checks the RESULTS_VALIDATION feature flag via AzureFeatureToggleService. If disabled, returns immediately with mode="disabled" and empty issue lists.
  • Iterates the ordered List<CelValidationRule> (sorted by priority ascending).
  • Calls rule.evaluate(request) on each rule and collects the returned List<ValidationIssue>.
  • Separates issues into errors and warnings lists.
  • Builds and returns DraftValidationResponse with isValid = errors.isEmpty().
3 Rule Discovery (Startup) — ValidationRuleAutoConfiguration

uk.gov.hmcts.cp.config.ValidationRuleAutoConfiguration

At application startup, scans classpath*:rules/DR-*.yaml and creates one CelValidationRule bean per file. Rules are sorted by priority and injected into DefaultValidationService as the validationRules bean. Adding a new rule requires only a new YAML file — no Java changes needed if an existing preprocessor type fits.

4 Rule Evaluation — CelValidationRule

uk.gov.hmcts.cp.services.rules.cel.CelValidationRule

For each rule, the following sequence runs:

  • Calls RuleOverrideService.findOverride(ruleId) to load any DB row from the validation_rule table (Caffeine-cached).
  • If the rule is disabled (YAML or DB), returns an empty list immediately.
  • Resolves the correct ValidationPreprocessor from PreprocessorRegistry using the YAML preprocessing.type field.
  • Calls preprocessor.preprocess(request, config) → Map<String, RuleEvaluationContext>.
  • For each context entry, evaluates every CEL condition (Steps 5–7).
  • For triggered conditions, constructs a ValidationIssue and applies SeverityCeiling.resolve().

The preprocessor transforms the raw request into a domain-specific context map ready for CEL evaluation. Two preprocessors are currently registered:

Step 6 — CEL Evaluation: CelExpressionEvaluator

uk.gov.hmcts.cp.services.rules.cel.CelExpressionEvaluator

  • Receives a CEL expression string (from the YAML conditions[].expression field) and the Map<String, Long> produced by context.toCelContext().
  • Computes a cache key from expression + "|" + sorted context keys.
  • Retrieves or compiles a typed Script (each variable declared as Int / long).
  • Executes the script against the context map and returns a boolean.
Example: expression "hasBothCount > 0" with context {hasBothCount: 0} → false → no issue generated.

Step 7 — Message Resolution: MessageTemplateResolver

uk.gov.hmcts.cp.services.rules.cel.MessageTemplateResolver

When a condition evaluates to true:

  • Retrieves the named offence ID list from context.getOffenceIdSet(condition.affectedOffenceSet).
  • Sorts offences by OffenceDisplayHelper.resolveOrderIndex() and formats each as "Offence N (URN:XXX)".
  • Joins with commas and "and": e.g. "Offence 1, Offence 2, and Offence 3".
  • Replaces ${offenceNumbers} and ${defendantName} placeholders in the YAML message template.
  • Returns the fully expanded human-readable message string.

Step 8 — Severity Ceiling: SeverityCeiling

uk.gov.hmcts.cp.services.rules.SeverityCeiling

Applies the DB override as a one-way ceiling. A WARNING override can cap an ERROR down; nothing promotes a WARNING to ERROR. 

resolve(yamlSeverity="ERROR",   dbSeverity="WARNING")  →  "WARNING"   ✓ capped down
resolve(yamlSeverity="WARNING", dbSeverity="ERROR")    →  "WARNING"   ✗ not promoted
resolve(yamlSeverity="ERROR",   dbSeverity=null)       →  "ERROR"       no override

Key Design Constraints

  • The DraftValidationRequest is read-only — no preprocessor or rule mutates it.
  • The DB severity ceiling is one-way — it can only cap downward, never promote a WARNING to ERROR.
  • Rules evaluate independently — there is no cross-rule shared state.
  • A rule failure is non-fatal — DefaultValidationService catches per-rule exceptions so one failing rule does not abort the rest.
  • Adding a new rule requires only a new YAML file if an existing preprocessor type covers the logic — no Java changes needed.
