# Upstream Contract Dependency (blocking prerequisite)

This feature cannot be implemented — the Java code will not compile — until the following
change lands in the **external** `api-cp-crime-hearing-results-validator` repository (the source
of `libs.api.hearing.results.validator`, currently pinned to `0.2.4` in
`gradle/libs.versions.toml`) and this repo's dependency is bumped to consume it. This mirrors the
exact pattern used for the `category` field added to `ResultLineDto` under DD-41656
(`specs/001-extended-test-disq-warning`).

## Required change (in `api-cp-crime-hearing-results-validator`)

Add a nullable field to `DefendantDto` in the OpenAPI spec (`openapi/openapi-spec.yml` in that
repo):

```yaml
DefendantDto:
  properties:
    # ... existing: defendantId, firstName, lastName, masterDefendantId
    dateOfBirth:
      type: string
      format: date
      nullable: true
      description: >-
        Defendant's date of birth. Nullable during the transition period while
        callers are updated to populate it.
```

Bump the published library's version once merged.

## Required change (in this repo, once the new version is available)

1. Bump the pin in `gradle/libs.versions.toml` to the new `api-cp-crime-hearing-results-validator`
   version.
2. Implement `AgeRestrictedImprisonmentPreprocessor` and `AgeRestrictedResultContext` per
   `data-model.md`, reading `DefendantDto.getDateOfBirth()`.
3. Add `src/main/resources/rules/DR-AGE-001.yaml` (draft above).
4. Unit + integration tests (see `tasks.md`, generated by `/speckit-tasks`).

## Callers that must populate the field (outside this repo, for the rule to ever fire)

Per the same pattern as `category` in DD-41656, at least one upstream caller of this validation
service's `/validate` endpoint must populate `dateOfBirth` on each `DefendantDto` it sends,
sourced from that caller's own domain model (the case/defendant record already holds a date of
birth in the case-management system — this is a data-plumbing change in the caller, not a new
data capture requirement). Until a caller does so, this rule remains permanently dormant
(fail-safe) for requests from that caller, per FR-011.
