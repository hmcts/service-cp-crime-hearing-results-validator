# Implementation Plan: Curfew and AAR Requirement End-Date Period Validation

**Branch**: `DD-41655-curfew-aar-end-date-validation` | **Date**: 2026-05-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/004-curfew-aar-end-date-validation/spec.md`

---

## Summary

Implement a new validation rule (`DR-COEW-002`) that detects when the end date recorded for a Curfew requirement (CUR, CURE, YRC2, YRC1) or an Alcohol Abstinence and Monitoring requirement (AAR) does not equal the date calculated from the entered period — blocking the clerk from saving until the discrepancy is corrected.

The rule requires:
1. **A blocking upstream library bump** (`api-cp-crime-hearing-results-validator` → `0.1.8`) to expose DURATION prompt nested structure.
2. **Three small framework enhancements** to `RuleEvaluationContext`, `MessageTemplateResolver`, and `CelValidationRule` that unlock `${expectedEndDate}` template resolution and `affectedDefendants` population for OFFENCE-level ERRORs.
3. **One new YAML rule file** (`DR-COEW-002.yaml`) with a single condition (`violationCount > 0`), authored first per Constitution Principle I.
4. **One new preprocessor** (`CurfewPeriodPreprocessor`) and **one new context record** (`CurfewPeriodContext`), each keyed per violation (one context per violation-type × defendant × offence triple).

**F-2 — Extend `MessageTemplateResolver` to resolve string variables**

File: `src/main/java/uk/gov/hmcts/cp/services/rules/cel/MessageTemplateResolver.java`

Extend `resolve()` to accept `Map<String, String> stringVariables` and apply each entry as a `${key}` → value replacement after the existing substitutions:

```java
public String resolve(String template, String defendantName,
                      List<String> affectedOffenceIds,
                      Map<String, OffenceDto> offenceMap,
                      List<String> allOffenceIds,
                      Map<String, String> stringVariables) {
    String result = resolve(template, defendantName,
                            affectedOffenceIds, offenceMap, allOffenceIds);
    for (Map.Entry<String, String> entry : stringVariables.entrySet()) {
        result = result.replace("${" + entry.getKey() + "}", entry.getValue());
    }
    return result;
}
```

Keep the original 5-arg `resolve()` intact (delegate to new method with `Map.of()`). All existing call sites continue to work.

Tests (TDD — write before implementation):
- String variable `${expectedEndDate}` is substituted correctly
- Multiple string variables substituted in one call
- Template with no string variable tokens is unchanged
- Empty `stringVariables` map leaves template unchanged

```yaml
rule:
  id: "DR-COEW-002"
  title: "Curfew and Alcohol Abstinence Requirement Period End Date Validation"
  description: >-
    Validates that the end date recorded for a Curfew requirement (CUR, CURE, YRC2, YRC1)
    equals the start date plus the entered period minus one day, and that the Until date
    for an Alcohol Abstinence and Monitoring requirement (AAR) equals the hearing date plus
    the number of abstinence days minus one day.
  priority: 4500
  enabled: true

  preprocessing:
    type: "curfew-period-check"
    communityOrderShortCodes:
      - COEW
      - COS
      - CONI
    yroShortCodes:
      - YROEW
      - YRONI
      - YROFEW
      - YROISS
      - YROINI
    curfewShortCodes:
      - CUR
      - YRC2
    curfewTagShortCodes:
      - CURE
      - YRC1
    alcoholAbstinenceShortCodes:
      - AAR

  conditions:
    - id: "AC1"
      name: "Requirement end date does not match entered period"
      expression: "violationCount > 0"
      severity: ERROR
      messageTemplate: >-
        The end date does not match the period entered. End date based on
        entered period: ${expectedEndDate}
      errorMessageTemplate: >-
        The end date does not match the period entered. End date based on
        entered period: ${expectedEndDate}
      affectedOffenceSet: "violatedOffenceIds"
```

**R-2 — Implement `CurfewPeriodContext`**

File: `src/main/java/uk/gov/hmcts/cp/services/rules/cel/CurfewPeriodContext.java`

```java
public record CurfewPeriodContext(
    String defendantId,
    String defendantName,
    String offenceId,
    long violationCount,         // always 1L
    String expectedEndDate       // "dd/MM/yyyy" formatted
) implements RuleEvaluationContext {

    @Override
    public Map<String, Long> toCelContext() {
        return Map.of("violationCount", violationCount);
    }

    @Override
    public List<String> getOffenceIdSet(final String setName) {
        return List.of(offenceId);  // any setName → singleton
    }

    @Override
    public List<String> allOffenceIds() {
        return List.of(offenceId);
    }

    @Override
    public String defendantId() {
        return defendantId;
    }

    @Override
    public Map<String, String> stringVariables() {
        return Map.of("expectedEndDate", expectedEndDate);
    }
}
```

Tests (TDD):
- `toCelContext()` returns `{"violationCount": 1L}`
- `getOffenceIdSet("violatedOffenceIds")` returns `[offenceId]`
- `getOffenceIdSet("anyOtherName")` also returns `[offenceId]` (no throw)
- `stringVariables()` returns `{"expectedEndDate": "30/07/2026"}`
- `defendantId()` returns the defendantId field
- `allOffenceIds()` returns `[offenceId]`

**R-3 — Implement `CurfewPeriodPreprocessor`**

File: `src/main/java/uk/gov/hmcts/cp/services/rules/cel/CurfewPeriodPreprocessor.java`

Key implementation points (full algorithm in `data-model.md`):

- `@Component`, `@Slf4j`
- `type()` returns `"curfew-period-check"`
- `parseDuration(line, promptRef)`:
  - Finds prompt by `promptRef`; requires `prompt.getChildPrompts()` non-empty
  - Reads child `promptRef` → maps to `ChronoUnit` (DAYS/WEEKS/MONTHS)
  - Reads child `promptValue` → parses to `long`
  - Returns `DurationValue(unit, quantity)` or null (with WARN log)
- `parseDate(line, promptRef)`:
  - Finds prompt by `promptRef`; reads `promptValue` as ISO-8601
  - Returns `LocalDate` or null (with WARN log)
- `parseInt(line, promptRef)`:
  - Finds prompt by `promptRef`; reads `promptValue` as integer string
  - Returns `Integer` or null (with WARN log)
- Expected end date for curfew: `startDate.plus(duration.quantity(), duration.unit()).minusDays(1)`
- Expected end date for AAR: `hearingDay.plusDays(days).minusDays(1)`
- Format with `DateTimeFormatter.ofPattern("dd/MM/yyyy", Locale.UK)`

Tests (TDD — extensive, see below):

**CUR violation scenarios:**
- CUR: end date matches calculated → no context emitted
- CUR: end date off by 1 day (too late) → context emitted with correct `expectedEndDate`
- CUR: end date off by 1 day (too early) → context emitted
- CUR: period in weeks → correct `plusWeeks` arithmetic
- CUR: period in months → correct `plusMonths` arithmetic
- CUR: missing `startDate` prompt → skip, WARN logged, no context
- CUR: missing `curfewPeriod` prompt → skip, WARN logged
- CUR: DURATION prompt has no childPrompts → skip, WARN logged
- CUR: unknown unit in child `promptRef` → skip, WARN logged

**CURE violation scenarios:**
- CURE: end date matches → no context
- CURE: end date mismatches → context with `expectedEndDate`
- CURE: missing `startDateOfTagging` → skip

**YRC2 and YRC1 scenarios** (same logic as CUR/CURE, different short codes):
- YRC2 under YROEW parent → violation detected
- YRC1 under YRONI parent → violation detected
- YRC2 under COEW parent → violation detected (CO parents also valid for CUR/YRC2 group)

**AAR scenarios:**
- AAR under COEW parent: Until date matches `hearingDay + days - 1` → no context
- AAR under COEW parent: Until date mismatches → context emitted
- AAR under YROEW parent (YRO-only) → AAR check SKIPPED (CO-only rule)
- AAR: `hearingDay` is null → skip, WARN logged
- AAR: missing `numberOfDaysToAbstainFromConsumingAnyAlcohol` → skip
- AAR: missing `until` → skip

**Multi-offence and multi-defendant scenarios:**
- Two offences, one violated → one context emitted for violated offence
- Two defendants, one violated → one context emitted for violated defendant
- Same offence has both CUR and CURE violations → two contexts (different keys)
- CUR violation on offence A, CURE violation on offence B → two contexts, distinct keys
